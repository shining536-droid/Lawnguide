- 서론: 장면형 3~4문장
- 본문: 단계별 3줄 구조
- 판례 1개, 3줄 공식
- FAQ 4~6개 기본, 필요한 경우만 8개
- 중간+하단 CTA 골드 강조
- 내부링크 5개

- 톤: 체계적 기준문서형
- **push 전 필수 체크리스트 (하나라도 빠지면 push 금지):**

  2. web/ 내 untracked .ts 파일이 있으면 반드시 `git add` (특히 spokes-batch*.ts 파일은 반드시 확인)
  3. spoke-pages.ts에서 import하는 파일이 전부 staged/committed 상태인지 확인
  4. `cd web && npx next build` 성공 확인
  5. 4단계 전부 통과 후에만 push

### 네이버 블로그 (하루 40개)
- 목적: 네이버 조회수 → 사이트 유입
- 제목: 질문형/상황형 ("자진퇴사했는데 실업급여 받을 수 있나요?")

  - 공감 도입 150~200자 + Tip 3개 × 400~450자 + 판례 150~200자 ≈ 1,500~1,800자

  - **생성 직후 반드시 `python check_blog_length.py {오늘날짜}` 실행** — 부족·공감도입 누락 파일은 즉시 재생성 후 다시 검증
- **블로그 본문 필수 구조 (4개 섹션, 하나라도 빠지면 발행 금지)**:
  1. **공감 도입 (150~200자)** — 본문 맨 앞, 2~3문장 공감 내러티브
     - 독자의 상황/감정에 공감하는 구어체 ("~막막하죠", "~걱정되시죠", "~얼마나 답답하실지")
     - 팁부터 바로 시작 금지 (이탈률 상승)
     - 예: "갑자기 해고 통보를 받으면 머릿속이 하얘지죠. '이게 합법인가? 내가 뭘 해야 하나?' 막막하기만 합니다. 하지만 지금 바로 챙겨야 할 것들이 있어요."

     - `**쉽게 설명하면**` — 1~2문장 (법조문/용어를 쉬운 말로)

  3. **판례 (150~200자)** — `### 실제로 이런 판례가 있었어요` + 3~4문장 (사건번호·법원·선고일·판시사항·시사점). 사건번호는 반드시 `kb/{domain}/cases.json`에 실존하는 것만 사용.

### 발행 전 필수 검증 (블로그/티스토리) — 8가지 자동 규칙
**`python check_blog_length.py {오늘날짜}` 가 아래 8개를 자동 검사합니다. publish 스크립트(`publish_lawnguide.py`, `publish_tistory.py`)는 실행 시 `validate_files()` 를 먼저 호출해 규칙 위반이면 발행 자체를 중단합니다.**

| # | 규칙 | 판정 | 기준 |
|---|------|------|------|
| 1 | **본문 길이** | 차단 | 1,500~1,800자 (프론트매터·마크다운 제외). <1,500 → 차단, >1,800 → 경고 |
| 2 | **공감 도입** | 차단 | 본문 맨 앞 ~ 첫 💡 Tip 전 영역이 ≥100자 AND 공감어미 포함 (죠/나요?/걱정/막막/답답…) |
| 3 | **Tip 구조** | 차단 | 블로그: Tip 3개 × 4 서브섹션(왜중요/쉽게설명/바로할일/흔한실수) 전부. 티스토리: 바로할일/흔한실수 2개 |
| 4 | **판례 실존** | 경고 | 본문 사건번호(20XX다/도/두/스/므/모/고합/구합)를 `kb/{domain}/cases.json` 판례ID와 대조, 미존재 시 경고 |
| 5 | **CTA 규칙** | 차단 | 네이버: 본문에 CTA 문구·`/chat?domain=`·`lawnguide.co.kr` 금지(publish 자동 삽입). 티스토리: 도메인별 CTA 1안 + `https://www.lawnguide.co.kr/...` 공식 URL 필수 |
| 6 | **제목 중복** | 경고 | 같은 도메인 기존 제목과 한글 어간 교집합 ≥3 이면 경고 + 유사 파일 3개 표시 |
| 7 | **톤 (구어체)** | 경고 | 판례 섹션 제외 후 문어체(입니다/됩니다/한다/이다) 비율 >50% + 샘플 10개 이상 |
| 8 | **가해자/무고 표현** | 차단/경고 | **차단**: "범인/범죄자/유죄입니다/처벌받습니다/처벌된다". **경고**: category=accused 인데 "혐의를 받고 있다면" / "해당할 소지가 있" 중 하나도 없음 |

**+ 구조 방어선**
- **기준 템플릿 인라인** — 위 "기준 템플릿" 코드블록(52~98행)이 절대 기준. 외부 레퍼런스 파일 의존성 없음
- **최근 블로그 복사 금지** — 드리프트 원인. 매일 위 인라인 템플릿으로 직접 재현
- **발행 전 검증 의무화** — publish 스크립트가 `validate_files()` 자동 호출, 규칙 위반 시 발행 중단 (dry-run 포함)

**수동 실행 명령어**
- 오늘 날짜: `python check_blog_length.py {YYYY-MM-DD}`
- 전체 스캔: `python check_blog_length.py --all`
- CI 게이팅: `python check_blog_length.py --strict` (차단 발생 시 exit 1)
- 특정 파일: `python check_blog_length.py --files <path> ...`

- sitemap, canonical, og:url, 메타태그 전부 www
- 내부링크는 상대경로(/guide/...) 유지
- 새 파일 생성 시 절대 URL 쓸 때 www 필수 확인

## 판례 연동 규칙
- SEO 스포크 생성 시 kb/{domain}/cases.json에서 실제 판례 1개 참조
- 네이버/티스토리 블로그 생성 시 판례 1개 참조
- 판례 형식: "사건번호(법원명, 선고일) - 판시사항 요약 + 판결요지 핵심"
- 판례 톤: "~사건에서 법원은 ~라고 판시했습니다"
- 판례 선택 시 해당 글 주제와 가장 관련 높은 판례 우선
- cases.json 없는 도메인은 기존 방식 유지

### 판례 재사용 규칙
- 같은 판례도 관점/유형이 다르면 재사용 가능
- 같은 관점+같은 유형 조합은 다른 판례 사용
- 예: 판례A를 피해자/준비서류형으로 썼으면, 가해자/어디부터형으로 재사용 OK
- 예: 판례A를 피해자/준비서류형으로 또 쓰는 건 금지
- SEO/네이버/티스토리가 같은 주제면 같은 판례 사용 가능 (채널별 분량·구조가 다르므로 중복 아님)

### 판례 사용 추적
- 스포크/블로그 생성 시 사용된 판례를 kb/used-cases.json에 기록
- 형식: {"판례ID", "domain", "perspective", "type", "channel", "slug"}
- 생성 전에 used-cases.json 확인해서 같은 판례ID+관점+유형 조합이면 스킵

### 판례 품질 필터
- 판결요지가 비어있는 판례는 사용 제외
- cases.json에서 판결요지가 있는 판례만 우선 선택

## 이지법 데이터 연동 규칙
- SEO/블로그 생성 시 kb/{domain}/legal_facts.json에서 관련 법조문 해설 참조
- kb/{domain}/amounts.json에서 정확한 수치/기간/금액 인용
- 판례 + 이지법 조합 예시:
  "형법 제347조에 따르면 사기죄는 10년 이하의 징역 또는 2천만원 이하의 벌금에 처한다(legal_facts).
  실제로 대법원 2024도1932 사건에서도~(판례)"
- 수치 인용 시 출처 명시: "~에 따르면" 또는 "~규정에 의하면"

## SEO 스포크 생성 규칙

핵심 원칙: "새 글은 새 키워드가 아니라, 새 상황을 다뤄야 한다."

### 생성 전 필수 체크
아래 항목 중 최소 2개 이상이 기존 스포크와 달라야 발행한다.
- 대상 사용자가 다른가
- 사건 단계가 다른가
- 세부유형이 다른가
- 증거 상태가 다른가
- 결과 행동이 다른가
- CTA 목적이 다른가
→ 2개도 안 달라지면 발행 보류, 기존 글을 보강한다.

### 생성 전 자동 검사 (출고 전 차단)
1. slug 중복 여부
2. 제목 유사도 (기존 스포크와 비교)
3. 핵심 키워드 충돌
4. 기존 검색의도 그룹 충돌
→ 충돌 시 새 스포크를 생성하지 않는다.

### 고유 존재 이유 정의
새 스포크마다 아래 1줄 정의를 반드시 작성한다.
"이 페이지는 [누구]의 [어떤 상황]에서 [어떤 준비]를 돕는 페이지다."
→ 기존 페이지와 구별되지 않으면 생성 금지.

### 본문 차별화
각 스포크는 아래 항목 중 최소 4개 이상이 기존 유사 스포크와 달라야 한다.
1. 첫 문단 (문제정의/장면)
2. 지금 할 일 3개
3. 준비서류 목록
4. CTA 문구 (피해자/가해자/무고별)
5. 확인된 사실 요약
6. 무료 상담 기관 안내

### FAQ 규칙
- 기본 FAQ 템플릿 복붙 금지
- 해당 페이지 사용자만 궁금할 질문 4~6개만 작성
- 다른 스포크 FAQ와 동일한 문장 사용 금지
- FAQ는 키워드 포획용이 아니라 해당 상황의 실제 질문이어야 한다

### 분기 기준 (같은 도메인 내)
새 스포크는 아래 분기축 중 최소 2개 이상이 달라야 생성한다.
- 관점: 피해자 / 가해자 / 무고
- 세부유형: 중고거래 / 투자 / 지인금전 / 보이스피싱 등
- 단계: 신고 전 / 조사 전 / 합의 전
- 핵심 쟁점: 차용증 유무, 연락두절, 초범 여부 등
→ 분기축이 1개 이하만 다르면 새 스포크 생성 금지.

### 블로그 vs SEO 역할 분리
- SEO 스포크: 구조화, 준비서류, 법적 쟁점, 절차, /chat 연결
- 네이버/티스토리: 공감형 도입, 사례 중심, 실수 방지, 핵심 요약, site로 넘기기
→ 블로그가 SEO 스포크의 축약판/복붙판이면 안 된다.

### 발행 후 관리 (4~8주 후)
- 노출 있음 → 유지/보강
- 노출 약함 + 고유성 높음 → 더 기다림
- 노출 0 + 기존과 유사 → 통합 후보
- 노출 0 + 얇음 → noindex/삭제 후보

### 운영 원칙
- 같은 검색의도를 노리는 제목만 바꾼 스포크는 만들지 않는다
- 상황, 증거, 단계, 행동, CTA가 실제로 달라질 때만 새 스포크를 만든다

## CTA 작성 규칙

### 기본 원칙
- "AI"는 CTA에 포함하되, 문장 맨 앞 반복 금지
- CTA 핵심은 항상 "문제 해결/상황 정리/대응 순서 확인"
- "무료"는 보조적으로만 사용, 남용 금지
- 추상형 문구 금지: "AI 무료상담 받기", "AI 진단 시작하기", "AI 분석 해보기"
- 좋은 방향: "{문제/상황}, AI로 {정리/점검/확인}하기"

### CTA 문구 3가지 유형
1. **문제 해결형**: "{문제/상황}, AI로 {정리/점검/확인}하기"
2. **가능성 판단형**: "{할 수 있는지/받을 수 있는지}, AI로 점검하기"
3. **준비자료/절차형**: "{서류/절차/체크리스트}, AI로 정리하기"
