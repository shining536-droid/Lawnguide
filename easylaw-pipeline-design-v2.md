# 이지법(easylaw.go.kr) 전면 수집 파이프라인 설계서 v2

> v1 대비 변경: 3중 수집 구조, legal_facts 중간 레이어 추가, 결과 카드 수동 검수 원칙,
> 프로토타입 우선 작업 순서, 질문 설계 원칙, 출력 카드 구조, 수요/수익 추정 포함


---

## 0. 사업 정의

이 서비스는 **"전월세 분쟁 상황에서 사용자의 현재 단계를 분류하고,
지금 해야 할 행동과 연결 경로를 빠르게 보여주는 검색형 생활법률 도우미"**다.

- 본질: 판단이 아니라 **정리와 연결**
- 수익: 변호사 프로필 광고 + CPA + 애드센스
- 구조: 베베플랜과 동일한 **입력 → 분기 → 결과 카드 → 연결**
- 확장: 전월세 → 체불임금 → 소액사건 → 개인회생(기초진단) → 상속(계산기)

### 수요/수익 추정

| 지표 | Year 1 (중앙) | Year 2 (중앙) |
|------|-------------|-------------|
| 월 UV | 10,000 | 50,000 |
| 애드센스 월수익 | 3~5만원 | 15~25만원 |
| 변호사 프로필 광고 | 0~30만원 | 100~300만원 |
| CPA | 거의 0 | 30~100만원 |
| **월 합계** | **5~35만원** | **150~400만원** |

법률 키워드는 CPC가 높아서 같은 UV 대비 수익이 베베플랜의 2~3배.

## 2. 수집 파이프라인 아키텍처 (v2)

v1과 가장 큰 차이: **3중 수집**, **legal_facts 중간 레이어**, **결과 카드 수동 검수**

```
[Phase 0: 프로토타입] ← v2 신규
  csmSeq=629 중 1페이지(예: 임차권등기명령)를
  PC본문 / 모바일 / 관련법령 3가지로 파싱
  → 셀렉터 확정, 데이터 구조 확정
  → 전체 수집 전에 반드시 이 단계를 거칠 것

[Phase 1: 3중 수집] ← v2 수정
  1-A. 목차 수집 (PC 사이드바 HTML)
       → config/sitemap-{csmSeq}.json

  1-B. 본문 수집 (PC + 모바일 교차)
       PC: 상세 본문, 테이블, 법령참조
       모바일: 구조가 단순해서 파싱 안전망
       → raw/{csmSeq}/{ccfNo}-{cciNo}-{cnpClsNo}.json

  1-C. 관련법령 수집 (별도 페이지)
       → raw/{csmSeq}/law-refs.json
       본문 파싱 실패해도 법률 근거 레이어는 유지

  1-D. 100문100답 수집
       → raw/{csmSeq}/qna.json

[Phase 2: 정규화]
  raw → normalized_units.json
  페이지 단위로 분리:
  → 텍스트 블록
  → 조건문 ("~한 경우", "~여부")
  → 수치 데이터 (금액, 기간, 비율)
  → 법령 참조 (법명 + 조항)
  → 절차 단계

[Phase 3: 법률 사실 추출] ← v2 신규
  normalized → kb/{domain}/legal_facts.json
  "규칙"이 아니라 "법률 사실/절차 사실" 단위
  예: "묵시적 갱신 시 임차인은 언제든 해지통고 가능, 3개월 후 효력"
  → 다른 도메인(상가, 체불임금)으로 확장할 때 재사용 가능

[Phase 4: 엔진 변환] (자동 생성 OK)
  legal_facts → engine 데이터
  → domains/{domain}/questions.json (질문 트리)
  → domains/{domain}/branches.json (분기 규칙)
  → domains/{domain}/question_map.json (질문↔법률사실 매핑)

[Phase 5: 결과 카드 생성] ← v2 수정: 반자동
  branches → results.json **초안** 자동 생성
  → ⚠️ 최종 문구는 반드시 수동 검수
  → results.json은 "사람이 만지는 파일"
  → 법령 근거는 자동, 행동 문구는 수동
```

## 5. 질문 설계 원칙

질문은 **법령을 물어보는 방식이 아니라 사용자 상황을 캐는 방식**이어야 한다.
총 8~12문항, 옵션 선택형, 1분 이내 완료.

### 질문 구조 (전월세/보증금 기준)

```
[1단계: 유형 구분]
Q1. 주택인가요, 상가인가요?                    → enum: 주택/상가
Q2. 계약이 종료되었나요?                        → boolean
Q3. 지금도 해당 주택에 거주 중인가요?            → enum: 거주중/퇴거예정/이미퇴거

[2단계: 위험 구분]
Q4. 보증금을 돌려받았나요?                      → enum: 전액/일부/미반환
Q5. 임대인과 연락이 되나요?                     → boolean
Q6. 보증보험(전세보증금반환보증)에 가입했나요?    → enum: 예/아니오/모름

[3단계: 권리보호 구분]
Q7. 전입신고를 했나요?                          → boolean
Q8. 확정일자를 받았나요?                        → boolean

[4단계: 특수상황 구분]
Q9. 전세사기가 의심되는 정황이 있나요?            → boolean
    (잠적, 다중피해, 허위설명 등)
Q10. 경매/공매 통지를 받았나요?                  → boolean

[선택] 금액/지역
Q11. 보증금 금액대는?                           → range
Q12. 주택 소재지는?                             → region (소액보증금 기준용)
```

```
  4. 셀렉터 확정 → config/selectors.json 작성

완료 조건: 1페이지에서 legal_fact 1개 이상 추출 가능 확인
```

## 13. 주의사항

1. **수집 속도**: 2초 이상 간격, 동시 요청 1개
2. **저작권**: easylaw는 영리 이용 포함 자유 활용 허용. 단 본문을 그대로 서비스에 노출하지 않음. 내부 KB로만 사용
3. **법적 근거**: 결과 카드에는 반드시 law.go.kr 법령 조문 링크를 **1차 출처**로 표시. easylaw는 2차 참고 링크
4. **갱신 주기**: amounts.json의 수치 데이터는 시행령 개정 시 업데이트 필요
5. **면책 문구**: "이 정보는 법적 효력을 갖는 유권해석이 아닙니다" **모든 결과 페이지에 필수 표시**
6. **결과 카드**: 자동 생성 후 **반드시 수동 검수**. 사용자에게 "조언"으로 읽힐 수 있으므로 문구에 특히 주의
7. **포지셔닝**: "법률 상담"이 아니라 "절차 안내 + 기관 연결". 변호사 프로필은 "광고 포함" 명시
8. **프로토타입 우선**: 전체 수집 전에 반드시 1페이지 프로토타입으로 셀렉터/구조 확정
